## 1. أثر قرار أن كل بيانات التطبيق تجريبية

بما أن بيانات التطبيق الحالية تجريبية ولا مشكلة بحذفها، فهذا يغير طريقة التنفيذ جذرياً للأفضل:

### 1.1 ما الذي أصبح مسموحاً فنياً؟

- يمكن تنفيذ تغييرات قاعدة بيانات كبيرة بدون بناء Migration معقدة لحفظ بيانات المستخدمين الحاليين.
- يمكن رفع `schemaVersion` وإعادة بناء قاعدة نظيفة عند المستخدم.
- يمكن تغيير بنية الترقيم والجداول المركزية دون محاولة تحويل كل السجلات القديمة بدقة.
- يمكن حذف/تجاهل البيانات التجريبية داخل SQLite وملفات AppData عند الانتقال للنسخة الجديدة.

### 1.2 ما الذي لا يعنيه هذا القرار؟

- لا يعني حذف ملفات الكود أو الخطط أو المحتوى من المستودع بلا سبب.
- لا يعني إهمال صحة Migration في الكود؛ حتى لو سنمسح البيانات الآن، يجب أن يبقى التطبيق سليماً عند فتح قاعدة قديمة.
- لا يعني كسر أول تشغيل أو نظام المستخدمين.
- لا يعني تنفيذ عشوائي؛ الدستور ما زال يفرض: كود + توثيق + رفع + فحص بناء.

### 1.3 التوصية العملية

نعتمد سياسة انتقال نظيفة:

```text
Development reset مقبول الآن
Production-safe migration لاحقاً قبل أول إصدار حقيقي للزبائن
```

أي أثناء التطوير الحالي يمكن للمستخدم حذف قاعدة البيانات المحلية التجريبية بعد كل تغيير جذري، لكن الكود نفسه يبقى منظماً وقابلاً للتطور.

## 2. الوضع الحالي للكود بعد الفحص

### 2.1 قاعدة البيانات الحالية

`schemaVersion = 3`

الجداول الأساسية الموجودة حالياً تغطي أغلب المجالات:

- إعدادات، سجل نشاط، نسخ احتياطي، ترقيم سنوي.
- أشخاص وجهات وأدوار أشخاص وفريق مكتب.
- وكالات قضائية.
- دعاوى ومراحل وجلسات وإجراءات دعوى.
- شركات ومراحل وشركاء ومديرين.
- عقود وأطراف وتنبيهات ونماذج ونسخ.
- إجراءات إدارية وخطوات.
- مهام يومية وسجل مهام.
- مستندات وروابط مستندات.
- مالية: اتفاقيات أتعاب، دفعات، مصاريف.
- نواقص وخط زمني.
- مكتبة قانونية وروابطها.
- أوامر عمل.

كما توجد جداول SQL مخصصة:

- المستخدمون والصلاحيات: `app_roles`, `app_users`, `role_permissions`, `user_sessions`, `audit_events`.
- إدخال الأرشيف القديم والورقي: `archive_batches`, `archive_items`, `agency_delegates`, `archive_reference_values`, `document_paper_metadata`.

### 2.2 المشكلة الأساسية

النظام الحالي يملك جداول كثيرة، لكنه لا يملك بعد «طبقة ملف مكتب» موحدة تقود كل شيء.

المطلوب المتفق عليه ليس فقط تحسين شاشات، بل بناء محور مركزي:

```text
Office File = ملف مكتب مستقل برقم داخلي غير قابل للتعديل
```

ثم تتفرع تحته:

```text

```

مع حالة واضحة:

```text
جاري / منتهي / مدخل من الأرشيف القديم
```

### 3.4 لماذا جدول وليس Provider فقط؟

لأن القرارات المطلوبة لا يمكن ضمانها بـ Provider فقط:

- رقم ملف غير قابل للتعديل.
- منع تكرار الرقم.
- الإغلاق الإداري.
- التفريق بين الجديد والأرشيف القديم.
- معرفة هل الملف منتهي لكن له مالية/أصول/إجراءات لاحقة.
- ربط مستند تسليم أو إغلاق.
- بناء التقارير الموحدة.

### 3.5 أثر كون البيانات تجريبية

## 4. الترقيم الداخلي حسب نوع الملف

### 4.1 الموجود حالياً

`SequenceService.generateNextInternalNumber()` يولد:

```text
2026/001
```

ويستخدم `YearlySequences` بمفتاح فريد على `year` فقط.

### 4.2 المطلوب النهائي

```text
دعوى/2026/0001
إجراء/2026/0001
عقد/2026/0001
شركة/2026/0001
وكالة/2026/0001
```

### 4.3 المطلوب فنياً

تغيير نموذج الترقيم إلى تسلسل حسب النوع والسنة:

```text
OfficeFileSequences
```

أو تعديل `YearlySequences` ليصبح:

```text
year
fileType / scope
prefix
lastNumber
```

مع Unique على:

```text
(year, fileType)
```

### 4.4 القرار الموصى به

بما أن البيانات تجريبية، الأفضل إنشاء جدول جديد واضح:

```text
OfficeFileSequences
```

بدلاً من ترقيع `YearlySequences` القديمة.

### 4.5 قواعد إلزامية

- الرقم يولد داخل Transaction.
- الرقم يحفظ في `OfficeFiles.fileNumber`.
- الرقم لا يعدل بعد إنشاء الملف، حتى للمالك.
- رقم ملف المكتب منفصل تماماً عن:
  - رقم أساس الدعوى.
  - رقم المعاملة.
  - رقم السجل التجاري.
  - رقم سند الوكالة.
  - رقم العقد الخارجي إن وجد.

  - النوع.
  - الحالة.
  - النواقص.
  - وجود أصل ورقي.
  - وجود مالية متبقية.
  - وجود إجراء لاحق.
  - تاريخ أقرب عمل.
- منع زر إنشاء جديد من شاشة الملفات، أو توجيهه فقط إلى مكتب العمل حسب القرار.

### 6.3 الإغلاق الإداري


يتحقق من:

- هل يوجد موعد/عمل مفتوح؟
- هل توجد مالية متبقية؟
- هل الأصل الورقي موجود أو مسلم؟
- هل يوجد إجراء لاحق بعد الإغلاق؟
- هل يوجد محضر تسليم؟

ثم يضع:

```text
OfficeFiles.status = closed
```

مع حقول الإغلاق.

فهم قوائم مرجعية لتوزيع العمل، وليسوا أدوار دخول افتراضية حالياً.

### 15.3 المطلوب فنياً

- استبدال/ترحيل `صاحب المكتب` إلى `مالك المكتب`.
- إنشاء أدوار نظامية افتراضية الثلاثة.
- إضافة hierarchy level للدور:

```text
owner = 100
manager = 80
seniorLawyer = 60
custom lower levels later
```

- منع:
  - المدير من تعيين مالك.
  - المستخدم من منح صلاحية لا يملكها.
  - حذف/تعطيل آخر مالك.
  - تخفيض صلاحية آخر مالك.
  - تعطيل نفسه بطريقة تقفل النظام.
- وضع صلاحيات المالية والإعدادات والتدقيق ضمن Sensitive.

### 15.4 أثر البيانات التجريبية

يمكن حذف مستخدمي الاختبار وإعادة إنشاء أول مالك وفق النظام الجديد، بدل ترحيل معقد.

## 16. المكتبة القانونية والمحتوى

### 16.1 القرار الحالي

محتوى المكتبة مؤجل بالكامل إلى ما بعد اكتمال البرمجة الأساسية.

### 16.2 الموجود حالياً

يوجد استيراد لمحتوى قانوني من `content/`، وفيه ملفات خام غير معتمدة.

### 16.3 المطلوب الآن

- لا نكمل جمع محتوى.
- لا نحذف الدستور الآن لأن آخر قرار كان تجاوز أمر الحذف والجولة الجديدة.
- نبقي البنية التقنية للمكتبة فقط.
- نمنع محتوى المكتبة من تعطيل خطة تحديث البرنامج.

## 17. مشاكل سلامة تقنية يجب إصلاحها قبل العمل الكبير

### 17.1 فهارس Drift مكررة

في `schema.dart` توجد فهارس مكررة بنفس الاسم:

```text
idx_case_parties_case
idx_case_phases_case
idx_case_sessions_case
idx_tasks_date
```

يجب تنظيفها قبل تشغيل `build_runner`.

### 17.2 عدم توفر Flutter/Dart في بيئة المساعد

لا يمكن تشغيل:

```text
flutter analyze
dart run build_runner build
```

في بيئة المساعد الحالية. لذلك أي تغيير Drift يحتاج من المستخدم تشغيل الأوامر على Windows وإرسال الخطأ فوراً.

### 17.3 دوال dispose معلقة

هناك دوال `dispose()` كثيرة تم تعليقها. ليست أولوية قبل بنية الملفات، لكنها دين تقني يجب تنظيفه تدريجياً.

### 17.4 البيانات التجريبية

بما أن البيانات تجريبية، يجب إضافة تعليمات واضحة للمستخدم عند أول تغيير قاعدة كبير:

```text
احذف قاعدة البيانات المحلية التجريبية أو استخدم زر reset development data إن أضفناه
```

## 18. الترتيب التنفيذي الأدق المقترح

### المرحلة صفر — تثبيت الأرضية

- إبقاء المستودع على `main` فقط.
- عدم تطبيق Stash القديم.
- إصلاح فهارس Drift المكررة.
- توثيق أن بيانات التطوير قابلة للحذف.
- تجهيز آلية Reset للبيانات التجريبية أو تعليمات حذف DB.

### المرحلة 1 — ملف المكتب الموحد والترقيم

- إضافة `OfficeFiles`.
- إضافة `OfficeFileSequences`.
- إضافة enum/ثوابت لأنواع الملفات وحالاتها.
- تعديل `SequenceService` أو إنشاء `OfficeFileNumberService`.
- إنشاء `OfficeFileRepository`.
- توليد أرقام:
  - دعوى/سنة/تسلسل.
  - إجراء/سنة/تسلسل.
  - عقد/سنة/تسلسل.
  - شركة/سنة/تسلسل.
  - وكالة/سنة/تسلسل.

### المرحلة 2 — ربط إنشاء العمل الجديد بالملف الموحد

- تعديل إنشاء الدعوى.
- تعديل إنشاء الإجراء.
- تعديل إنشاء العقد.
- تعديل إنشاء الشركة.
- تعديل إنشاء الوكالة.
- كل إنشاء جديد يتم Transaction واحدة: OfficeFile + Entity + Timeline/Audit.

### المرحلة 3 — شاشة ملفات المكتب

- جعل `FilesScreen` تقرأ من `OfficeFiles`.
- اعتماد جارية/منتهية.
- إضافة مؤشرات:
  - نواقص.
  - مالية.
  - أصل ورقي.
  - أقرب عمل.
  - إجراءات لاحقة.
- منع إنشاء جديد من شاشة الملفات.

### المرحلة 4 — الإغلاق الإداري

- إضافة شاشة/حوار إغلاق ملف.
- فحص المالية والأصول والأعمال المفتوحة.
- حفظ سبب الإغلاق وملخصه ومحضر التسليم.
- منع إدخال أعمال عادية على ملف منتهي إلا كإجراء لاحق مصرح.

### المرحلة 5 — مركز اليوم والغد

- إعادة ضبط دلالات اليوم والغد.
- إضافة غير المكتمل.
- إعادة ضبط حالة توزيع العمل.
- ربط أوامر العمل بـ OfficeFile.
- تحسين واتساب.

### المرحلة 6 — الصلاحيات

- إعادة تسمية/إنشاء الأدوار الافتراضية.
- hierarchy.
- منع التصعيد ومنح ما لا يملك المستخدم.
- حماية المالية والإعدادات والتدقيق.

### المرحلة 7 — المستندات والمالية

- ربط المستندات والمالية بـ OfficeFile.
- إظهار مؤشرات الملفات.
- تحضير الصندوق لاحقاً بعد ثبات الملف الموحد.

### المرحلة 8 — المكتبة والتصميم النهائي

- تأجيل محتوى المكتبة.
- تأجيل التجميل النهائي للواجهات إلى ما بعد اكتمال البرمجة الأساسية.

## 19. ما لا يجب لمسه الآن

- لا جمع محتوى قانوني جديد.
- لا حذف ملفات مكتبة قانونية بسبب موضوع الدستور حالياً.
- لا إعادة تصميم شكلي كامل الآن.
- لا إدخال نظام ترخيص تجاري قبل اكتمال قلب البرنامج.
- لا شبكة داخلية/PostgreSQL الآن.
- لا ترحيل معقد لحفظ بيانات الاختبار.

## 20. القرار الفني النهائي المقترح قبل بدء الكود

أقترح اعتماد هذه القاعدة للتنفيذ:

```text
نبني OfficeFiles + OfficeFileSequences أولاً، ونعتبرهما قلب النسخة الجديدة.
كل شيء آخر يرتبط بهما تدريجياً.
بما أن البيانات تجريبية، نسمح بقاعدة بيانات نظيفة بعد أول تغيير جذري.
```

إذا تم اعتماد هذا القرار، فالخطوة التالية ليست كود عشوائي، بل كتابة:

```text
docs/FINAL_IMPLEMENTATION_ROADMAP.md
```

وفيها لكل مرحلة:

- الملفات التي ستتغير.
- جداول Drift الجديدة.
- الـ Repositories المطلوبة.
- الشاشات المتأثرة.
- طريقة الفحص على Windows.
- أوامر `build_runner` و `flutter analyze` التي سيشغلها المستخدم.
