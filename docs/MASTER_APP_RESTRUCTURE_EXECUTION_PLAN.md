# الخطة التنفيذية الرئيسية لإعادة هيكلة تطبيق ميزان

> **الغرض:** تجميع كل القرارات المعتمدة في خطة تنفيذ واحدة قبل تعديل الكود، حتى يتم التنفيذ بدقة وبلا عودة لتعديلات متضاربة.  
> **النطاق:** إعادة هيكلة الشريط الجانبي، ملفات المكتب، مكتب العمل، الأرشفة، المكتبة القانونية، النماذج، المالية، التقارير، إدارة المكتب، ثم المساعد الذكي لاحقاً.  
> **شرط التنفيذ:** لا تُعتبر أي مرحلة مكتملة إلا بعد التحليل، الاختبار، التوثيق، والرفع.

---

## 1. الشروط الصارمة

1. لا تغيير في أي منطق خارج نطاق المرحلة المنفذة.
2. لا حذف لجداول أو بيانات حالية قبل وجود بديل مستقر.
3. لا حذف لمسارات قديمة فوراً؛ تُخفى من الواجهة أو تُحوّل لاحقاً بـ Redirect عند الحاجة.
4. كل تعديل يجب أن يحترم الصلاحيات وسجل المسؤولية.
5. لا إدخال شاشة جديدة بلا دور واضح في الهيكل النهائي.
6. لا تنفيذ المساعد الذكي قبل استقرار الهيكل.
7. كل مرحلة تنتهي بـ:

```bash
flutter clean
flutter pub get
flutter analyze
flutter run -d windows
```

قدر الإمكان في بيئة المستخدم، مع إصلاح أي خطأ فوراً.

## 2. الشريط الجانبي النهائي المعتمد

الشريط الجانبي النهائي بعد إعادة الهيكلة:

```text
مكتب العمل
ملفات المكتب
المكتبة القانونية
النماذج القانونية
المالية والصندوق
التقارير والكشوف
إدارة المكتب
```

### 2.1 مكتب العمل

يضم لاحقاً:

```text
اليوم
الغد
الأسبوع
التقويم
أوامر العمل
```

لكن التنفيذ المرحلي يجب ألا ينشئ روابط غير موجودة قبل بناء الشاشات.

### 2.2 ملفات المكتب

بديل `الشؤون القانونية` وبديل شاشات الأشخاص والوكالات والأرشيف الفيزيائي المستقلة.

يقسم إلى:

```text
الملفات الجارية
الملفات المنتهية
```

وداخلهما:

```text
الدعاوى
الإجراءات والمعاملات
العقود
الشركات
الوكالات
```

### 2.3 المكتبة القانونية

قسم مستقل للمراجع القانونية:

```text
القوانين السورية
اجتهادات محكمة النقض
الهيئة العامة
مجلة المحامون
أبحاث ومذكرات
كتب ومراجع
```

### 2.4 النماذج القانونية

قسم مستقل للقوالب:

```text
استيراد نموذج
إنشاء نموذج
إدارة النماذج
توليد مستند من نموذج
```

### 2.5 المالية والصندوق

قسم مستقل بسبب أهميته وحساسيته:

```text
لوحة مالية
اتفاقيات الأتعاب
سندات القبض
المصاريف
ذمم الموكلين
صندوق المكتب
التقارير المالية
```

### 2.6 التقارير والكشوف

يشمل الطباعة المركزية ولا يوجد `مركز الطباعة` كقسم مستقل:

```text
كشوف الدعاوى
كشوف الجلسات
كشوف النواقص
كشوف أوامر العمل
كشوف المالية
لائحة عمل اليوم والغد
تصدير PDF
```

### 2.7 إدارة المكتب

بديل `إعدادات النظام` كتسمية للمحامي:

```text
بيانات المكتب
المستخدمون والصلاحيات
سجل المسؤولية
النسخ الاحتياطي
القوائم المرجعية
```

## 4. ترتيب التنفيذ البرمجي المرحلي

### المرحلة 0 — تثبيت خط الأساس

- التأكد من بناء آخر نسخة.
- إصلاح أي أخطاء حالية قبل الهيكلة.
- عدم تعديل سلوك وظيفي.

### المرحلة 1 — إعادة تسمية وتنظيم الشريط الجانبي مؤقتاً دون كسر المسارات

الهدف:
- تغيير التسميات بما يوافق الهيكل النهائي.
- إبقاء المسارات القديمة صالحة.
- عدم إنشاء شاشات جديدة كبيرة بعد.

أعمال المرحلة:
- `الشؤون القانونية` → `ملفات المكتب`.
- `الإجراءات الإدارية` → `الإجراءات والمعاملات`.
- إزالة مجموعة `السجلات والجهات` من الشريط الجانبي الرئيسي أو إخفاؤها مرحلياً حسب توفر البدائل.
- `الموارد والمستندات` تُفكك مرحلياً.
- `الإدارة والتقارير` تُفكك إلى `المالية والصندوق` و`التقارير والكشوف` و`إدارة المكتب`.

### المرحلة 2 — بناء شاشة ملفات المكتب الموحدة

الهدف:
- شاشة واحدة بفلاتر:

```text
جارية / منتهية / تحتاج استكمال
دعاوى / إجراءات / عقود / شركات / وكالات
تصنيفات فرعية لكل نوع
```

### المرحلة 3 — دمج الأشخاص والوكالات داخل ملفات المكتب والمعالجات

- لا تبقى الأشخاص والوكالات كأقسام مستقلة.
- تبقى كبيانات أطراف داخل الملفات.
- تعديل معالجات الإضافة لتدعم اختيار/إضافة الأطراف والوكالات داخلياً.

### المرحلة 4 — مكتب العمل اليومي

- تحويل لوحة اليوم إلى مكتب العمل:

```text
اليوم = إدخال النتائج
الغد = التحضير والتوزيع
الأسبوع = رؤية قريبة
التقويم = عرض زمني
```

- دعم المهام المخصصة وقوالب نتائجها لاحقاً.

### المرحلة 5 — مركز إدخال الأرشيف القديم

- دفعات إدخال.
- أرشيف ورقي/إلكتروني/CSV.
- تمييز الجاري والمنتهي.
- صندوق غير مصنف.
- ملفات جارية تحتاج استكمال.

### المرحلة 6 — المكتبة القانونية

- استقلالها عن المستندات.
- تجهيزها للقوانين والاجتهادات والمراجع.

### المرحلة 7 — النماذج القانونية

- استيراد نموذج.
- إنشاء نموذج.
- توليد مستند من نموذج.

### المرحلة 8 — التقارير والكشوف

- دمج الطباعة المركزية.
- إبقاء طباعة الملفات داخل ملفاتها.

### المرحلة 9 — المالية والصندوق

- إعادة هيكلة مالية الملف وذمم الموكلين والصندوق.

### المرحلة 10 — إدارة المكتب والقوائم المرجعية

- إدارة القوائم المرجعية بإضافة/تعديل/تعطيل/حذف آمن.

### المرحلة 11 — الاختبار الشامل

- اختبار كل شاشة وكل وظيفة.
- اختبار الصلاحيات.
- اختبار Audit.

### المرحلة 12 — المساعد الذكي

- بعد استقرار كل شيء فقط.

## 5. سياسة التنفيذ العملية

كل مرحلة تُنفذ كفرع منطقي داخلي وcommit واضح، ولا نخلط أكثر من نطاق كبير في commit واحد إلا عند الضرورة.

> ملاحظة: هذه مرحلة تنظيم تنقل وتسميات فقط. لم يتم حذف أي جداول أو مسارات قديمة، ولم يتم نقل وظائف الأشخاص/الوكالات/الأرشيف الفيزيائي نهائياً بعد. ذلك سيتم في المراحل اللاحقة حسب الخطة.

## سجل تنفيذ فعلي — المرحلة 2: بداية شاشة ملفات المكتب الموحدة 2026-07-15

تم تنفيذ خطوة أولى من المرحلة الثانية دون حذف الشاشات القديمة:

- توسيع مزود `uiFilesProvider` ليجمع ملفات المكتب من مصادر حقيقية متعددة:
  - الدعاوى
  - العقود
  - الشركات
  - الإجراءات والمعاملات
  - الوكالات
- إعادة تسمية شاشة `/files` إلى `ملفات المكتب`.
- تحديث تبويب `عاملة` إلى `جارية` بما يوافق مصطلحات الهيكلة الجديدة.
- جعل بطاقة الملف تفتح المسار الحقيقي حسب نوع الملف:
  - دعوى → `/cases/:id`
  - عقد → `/contracts/:id`
  - شركة → `/companies/:id`
  - إجراء → `/procedures/:id`
  - وكالة → `/poa/:id`
- إبقاء تبويبات الحالة الحالية مؤقتاً لحين بناء الواجهة النهائية ذات طبقات:
  - جارية / منتهية / تحتاج استكمال
  - النوع
  - التصنيف الفرعي

> هذه خطوة تأسيسية لشاشة ملفات المكتب، وليست النسخة النهائية الكاملة. الهدف الحالي هو توحيد مصادر الملفات وفتح كل ملف من مكان واحد دون كسر المسارات الحالية.

  - المعروض

تمت إضافة طبقة التصنيف الفرعي إلى شاشة ملفات المكتب:

- إضافة حقل `subCategory` إلى نموذج `FileItem`.
- تعبئة التصنيف الفرعي من المصدر الحقيقي:
  - الدعاوى: نوع الدعوى.
  - العقود: نوع العقد.
  - الشركات: الشكل القانوني.
  - الإجراءات: نوع الإجراء.
  - الوكالات: نوع الوكالة.
- إضافة فلتر `كل التصنيفات` في شاشة ملفات المكتب، ويتغير حسب نوع الملف المختار.
- إظهار التصنيف الفرعي كبطاقة صغيرة داخل بطاقة الملف.

هذا يطبق مبدأ الطبقات المتفق عليه:

```text
الحالة ← نوع الملف ← التصنيف الفرعي
```

دون إنشاء شاشات منفصلة أو تكرار واجهات.

### سجل تنفيذ فعلي — ضبط شاشة عمل جديد وفق الهيكل الجديد 2026-07-15

تمت مواءمة شاشة `عمل جديد` مع قرار دمج الأشخاص والمستندات داخل ملفات المكتب والمعالجات:

- إزالة بطاقة `شخص / جهة` كعمل مستقل، لأن إضافة الأشخاص والجهات يجب أن تتم من داخل ملفات الدعوى/الإجراء/العقد/الشركة/الوكالة أو عبر البحث الداخلي.
- إزالة بطاقة `مستند` كعمل مستقل، لأن المستندات أصبحت طبقة مرفقات داخل ملفات المكتب ومركز الأرشيف وليست مسار عمل مستقل.
- إعادة تسمية `وكالة` إلى `ملف وكالة` وربطها بسياق ملفات المكتب.
- تحديث تمييز الشريط الجانبي بحيث تندرج المسارات القديمة الداخلية مثل `/persons` و`/archive` تحت `ملفات المكتب` إذا فُتحت مباشرة.

### سجل تنفيذ فعلي — توحيد مسارات النماذج والتقارير 2026-07-15

تم تنفيذ خطوة تنظيمية إضافية ضمن إعادة الهيكلة:

- تحويل المسار القديم `/contracts/templates` إلى Redirect نحو `/templates` حتى لا تبقى قوالب العقود كمسار مستقل عن `النماذج القانونية`.
- حماية المسار القديم بنفس صلاحية `templates.view`.
- تحديث عنوان شاشة `البحث والتقارير` إلى `التقارير والكشوف`.
- تحديث تبويب `التقارير` إلى `الكشوف والتقارير`.
- تحديث شاشة الطباعة القانونية إلى `الطباعة والتصدير` ضمن مفهوم التقارير والكشوف.

> هذه الخطوة تحفظ التوافق مع المسارات القديمة دون إبقاء التسمية القديمة أو ربط النماذج بالعقود فقط.

### سجل تنفيذ فعلي — مواءمة شاشة إدارة المكتب 2026-07-15

تمت مواءمة شاشة الإعدادات مع تسمية `إدارة المكتب` المعتمدة في الشريط الجانبي:

- تغيير عنوان الشاشة إلى `إدارة المكتب`.
- إعادة ترتيب التبويبات حسب الأولوية الإدارية:
  - بيانات المكتب
  - الأمان
  - المستخدمون والصلاحيات
  - سجل المسؤولية
  - النسخ الاحتياطي
  - القوائم المرجعية
- إخفاء تبويب `سجل النشاط` القديم من الواجهة الرئيسية، مع إبقاء الكود الداخلي مؤقتاً إلى حين استقرار سجل المسؤولية النهائي.

> الهدف هو جعل إدارة المكتب مركزاً واضحاً للضبط والإدارة، وليس شاشة إعدادات تقنية متفرعة.

### سجل تنفيذ فعلي — توضيح تبويبي ملفات المكتب ومعالج الأرشيف 2026-07-17

تمت مواءمة واجهة `ملفات المكتب` مع القاعدة العملية الجديدة:

- التبويب الرئيسي أصبح واضحاً بين:
  - `الملفات الجارية`: تؤثر على مكتب العمل والمواعيد القادمة.
  - `الملفات المنتهية`: للحفظ والبحث فقط.
- تم تعديل الشريط الجانبي ليعرض هذين المدخلين مباشرة تحت `ملفات المكتب` قبل `إدخال الأرشيف القديم`.
- تم بناء بداية معالج إدخال الأرشيف القديم وفق مسار: حالة الأرشيف ← نوع الملف ← التصنيف التفصيلي ← الوثائق ← شاشة الإدخال الرسمية.

> الهدف هو منع خلط الأرشيف المنتهي بالملفات الحية التي يجب أن تغذي مكتب العمل.

### سجل تنفيذ فعلي — ربط معالج الأرشيف بالشاشات الرسمية 2026-07-17

تمت إضافة طبقة `ArchiveEntryContext` لتمرير سياق الأرشيف من معالج الإدخال إلى شاشات الإنشاء الرسمية. الهدف هو أن يشعر المستخدم أنه يعمل ضمن أرشيف جارٍ أو منتهٍ، مع الحفاظ على مسارات الحفظ الأصلية وعدم إنشاء منطق موازٍ.

الشاشات التي أصبحت تستقبل سياق الأرشيف:

- الدعوى.
- الشركة.
- الإجراء / المعاملة.
- العقد.

الملف الجاري يبقى قابلاً لتوليد مواعيد في مكتب العمل من نفس حقول الشاشة الرسمية، أما الملف المنتهي فيظهر تنبيه واضح أنه للحفظ والبحث فقط.

- المتابعة في الويزارد تكون تذكيراً واحداً فقط، وتضاف المتابعات الأخرى لاحقاً من ملف العقد الجاري.
- الأتعاب اختيارية ومنفصلة عن قيمة العقد بين الأطراف.

### قرار مصطلحي ملزم — رقم ملف المكتب ورقم أساس الدعوى 2026-07-17

تم الاتفاق على قاعدة مصطلحية وتشغيلية مهمة يجب تطبيقها في كل شاشات الملفات والدعاوى والتقارير:

```text
رقم ملف المكتب ≠ رقم أساس الدعوى
```

#### رقم ملف المكتب

هو الرقم الداخلي الذي ينشئه التطبيق أو المكتب لإدارة الملف داخل المكتب.

أمثلة:

```text
2026/001
C-2026-015
```

استخدامه:

- البحث الداخلي.
- ترتيب ملفات المكتب.
- الطباعة والتقارير.
- ربط المستندات.
- ربط الأتعاب والمصاريف.
- أوامر العمل.
- الأرشيف الداخلي.

التسمية المعتمدة في الواجهة:

```text
رقم ملف المكتب
```

أو عند الضرورة التقنية:

```text
الرقم الداخلي
```

لكن لا يسمى `رقم الدعوى` إذا كان المقصود رقم ملف المكتب.

#### رقم أساس الدعوى

هو الرقم القضائي الصادر عن المحكمة أو المرجع القضائي.

أمثلة:

```text
أساس 120 لعام 2026
أساس استئناف 44 لعام 2027
```

التسمية المعتمدة:

```text
رقم الأساس
سنة الأساس
```

ويجب ربطه بالمرحلة القضائية عند تعدد المراحل:

```text
أساس الصلح
أساس البداية
أساس الاستئناف
أساس التنفيذ
```

#### النقض والمخاصمة

في النقض والمخاصمة قد لا تكون التسمية العملية دائماً `أساس`، لذلك تعتمد تسميات مثل:

```text
رقم الطعن
سنة الطعن
رقم المرجع القضائي
```

حسب المرحلة.

#### أثر القرار على التنفيذ

- يجب مراجعة كل التسميات القديمة التي تقول `رقم الدعوى`.
- إذا كان المقصود رقم المكتب، تستبدل بـ `رقم ملف المكتب`.
- إذا كان المقصود رقم المحكمة، تستبدل بـ `رقم الأساس`.
- في شاشة الدعوى يجب فصل بيانات المكتب عن بيانات المحكمة.
- يجب أن تكون مراحل الدعوى قادرة على حمل أرقام أساس مختلفة لكل مرحلة.

> هذا القرار يمنع الخلط بين إدارة المكتب وبين القيود القضائية الرسمية، وهو شرط أساسي قبل إعادة بناء شاشة الدعوى الجارية.

### قرار تصميمي — شاشة الملفات الجارية 2026-07-17

تم الاتفاق على هيكلة شاشة `الملفات الجارية` بالشكل التالي:

- الشاشة هي لوحة موحدة لكل الملفات الحية التي يعمل عليها المكتب حالياً.
- لا تحتوي مدخلاً لإنشاء أرشيف أو إنشاء ملف جديد؛ الإنشاء الجديد يبقى من `مكتب العمل > إنشاء جديد` فقط.
- تحتوي على بحث شامل في:
  - رقم ملف المكتب.
  - الموضوع / العنوان.
  - الموكل أو الجهة.
  - الخصم عند توفره.
  - المحكمة أو الجهة.
  - رقم الأساس أو المرجع.
- تحتوي فلاتر نوع الملف:
  - كل الملفات.
  - دعاوى.
  - إجراءات / معاملات.
  - شركات.
  - عقود.
  - وكالات.
- تحتوي فلاتر تشغيلية:
  - لديها موعد قريب.
  - متأخرة.
  - تحتاج استكمال.
  - لديها أوامر عمل مفتوحة.
  - بلا مستندات.
- تحتوي مؤشرات علوية:
  - المعروض.
  - دعاوى.
  - إجراءات.
  - شركات.
  - عقود.
  - وكالات.
  - متأخرة.
  - تحتاج استكمال.
- طريقة العرض المعتمدة مبدئياً: بطاقات ملفات مع تصدير/طباعة، وتأجيل عرض الجدول إلى مرحلة لاحقة عند الحاجة.
- ترتيب العرض الافتراضي:
  - الملفات المتأخرة أو الأقرب موعداً أولاً.
  - ثم حسب الموعد القادم.
  - ثم آخر تحديث.
- كل بطاقة ملف تعرض:
  - رقم ملف المكتب.
  - نوع الملف.
  - التصنيف الفرعي.
  - الموضوع.
  - الموكل أو الجهة الأساسية.
  - المحكمة/الجهة أو المرجع.
  - رقم الأساس أو المرجع عند توفره.
  - الموعد القادم والإجراء القادم عند توفرهما.
  - النواقص.
  - عدد المستندات.
  - آخر تحديث.
- الأزرار السريعة داخل البطاقة تكون محدودة:
  - فتح الملف.
  - إضافة أمر عمل.
  - إجراء تشغيلي سريع بحسب النوع عند الحاجة، مثل إدخال نتيجة جلسة للدعوى أو تحديث خطوة للإجراء.
- التعديل الكامل يتم من شاشة تفاصيل الملف المتخصصة حسب نوعه، وليس من بطاقة الملفات الجارية.
- فتح الملف ينتقل إلى شاشة تفاصيله الأصلية:
  - دعوى ← شاشة الدعوى.
  - إجراء ← شاشة الإجراء.
  - شركة ← شاشة الشركة.
  - عقد ← شاشة العقد.
  - وكالة ← شاشة الوكالة.

> هذه الهيكلة تعتمد قائمة موحدة للملفات الجارية مع تفاصيل متخصصة لكل نوع ملف، وهي الأقرب للتنفيذ الحالي والأفضل لتجنب بناء شاشة ضخمة واحدة لكل الأنواع.

### قرار تصميمي — شاشة الدعوى الجارية Workspace 2026-07-17

تم الاتفاق على هيكلة شاشة الدعوى الجارية وفق المبادئ التالية:

- تبويبات متخصصة لا شاشة طويلة واحدة.
- رقم ملف المكتب منفصل تماماً عن رقم الأساس القضائي.

  - قيد التحضير.
  - مسجلة.
  - قيد التبليغ.
  - قيد المحاكمة.
  - بانتظار قرار.
  - بانتظار تنفيذ.
  - معلقة.
  - مؤجلة.

### قرار تصميمي — آثار الإغلاق والإجراءات اللاحقة للدعاوى 2026-07-18

تم اعتماد القاعدة العملية التالية لشاشة الدعوى الجارية والمنتهية:

```text
شطب الدعوى = مسار تجديد الدعوى، وليس إغلاقاً نهائياً للحق.
رد الدعوى شكلاً = يفتح استئناف/نقض/دعوى جديدة مرتبطة حسب المرحلة، ولا يعامل كخسارة موضوعية تلقائية.
التخلي عن الملف / تسليمه = إغلاق علاقة المكتب بالملف مع دعم لا مانع وإمكانية إعادة فتح الملف لاحقاً إذا عاد للمكتب.
التنازل عن الحق = إغلاق نهائي حساس يتطلب تحذيراً وصلاحية مناسبة ومستنداً أو سبباً واضحاً.
صدور حكم أو قرار = إنهاء مرحلة وفتح التصعيد المناسب: استئناف، نقض، مخاصمة، تنفيذ، أو إغلاق نهائي.
```

أثر القرار على التنفيذ:

- زر الدعوى الأساسي يكون `إغلاق / إنهاء مرحلة` لا `إغلاق الدعوى` فقط.
- الملفات المنتهية يجب أن تحتوي قسماً أو تبويباً باسم `إجراءات لاحقة / طعون`.
- زر `تجديد الدعوى` يظهر عند الشطب أو الحالات الشكلية التي تقبل التجديد أو إعادة القيد بحسب السياق.
- أزرار الاستئناف والنقض والمخاصمة والتنفيذ تظهر بحسب المرحلة والاختصاص، لا كأزرار عامة دائماً.
- التخلي عن الملف يدعم إنشاء وطباعة `لا مانع` قابل للتعديل.
- إعادة فتح ملف منتهٍ إلى جارٍ ممكنة بصلاحية وسبب وسجل أحداث.
