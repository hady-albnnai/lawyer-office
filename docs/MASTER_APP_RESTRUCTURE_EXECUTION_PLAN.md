# الخطة التنفيذية الرئيسية لإعادة هيكلة تطبيق ميزان

> **الغرض:** تجميع كل القرارات المعتمدة في خطة تنفيذ واحدة قبل تعديل الكود، حتى يتم التنفيذ بدقة وبلا عودة لتعديلات متضاربة.  
> **النطاق:** إعادة هيكلة الشريط الجانبي، ملفات المكتب، مكتب العمل، الأرشفة، المكتبة القانونية، النماذج، المالية، التقارير، إدارة المكتب، ثم المساعد الذكي لاحقاً.  
> **شرط التنفيذ:** لا تُعتبر أي مرحلة مكتملة إلا بعد التحليل، الاختبار، التوثيق، والرفع.

---

## 1. الشروط الصارمة

1. لا تغيير في أي منطق خارج نطاق المرحلة المنفذة.
2. لا حذف لجداول أو بيانات حالية قبل وجود بديل مستقر.
3. لا حذف لمسارات قديمة فوراً؛ تُخفى من الواجهة أو تُحوّل لاحقاً بـ Redirect عند الحاجة.
4. كل تعديل يجب أن يحترم الصلاحيات وسجل المسؤولية.
5. لا إدخال شاشة جديدة بلا دور واضح في الهيكل النهائي.
6. لا تنفيذ المساعد الذكي قبل استقرار الهيكل.
7. كل مرحلة تنتهي بـ:

```bash
flutter clean
flutter pub get
flutter analyze
flutter run -d windows
```

قدر الإمكان في بيئة المستخدم، مع إصلاح أي خطأ فوراً.

## 2. الشريط الجانبي النهائي المعتمد

الشريط الجانبي النهائي بعد إعادة الهيكلة:

```text
مكتب العمل
ملفات المكتب
المكتبة القانونية
النماذج القانونية
المالية والصندوق
التقارير والكشوف
إدارة المكتب
```

### 2.1 مكتب العمل

يضم لاحقاً:

```text
اليوم
الغد
الأسبوع
التقويم
أوامر العمل
```

لكن التنفيذ المرحلي يجب ألا ينشئ روابط غير موجودة قبل بناء الشاشات.

### 2.2 ملفات المكتب

بديل `الشؤون القانونية` وبديل شاشات الأشخاص والوكالات والأرشيف الفيزيائي المستقلة.

يقسم إلى:

```text
الملفات الجارية
الملفات المنتهية
```

وداخلهما:

```text
الدعاوى
الإجراءات والمعاملات
العقود
الشركات
الوكالات
```

### 2.3 المكتبة القانونية

قسم مستقل للمراجع القانونية:

```text
القوانين السورية
اجتهادات محكمة النقض
الهيئة العامة
مجلة المحامون
أبحاث ومذكرات
كتب ومراجع
```

### 2.4 النماذج القانونية

قسم مستقل للقوالب:

```text
استيراد نموذج
إنشاء نموذج
إدارة النماذج
توليد مستند من نموذج
```

### 2.5 المالية والصندوق

قسم مستقل بسبب أهميته وحساسيته:

```text
لوحة مالية
اتفاقيات الأتعاب
سندات القبض
المصاريف
ذمم الموكلين
صندوق المكتب
التقارير المالية
```

### 2.6 التقارير والكشوف

يشمل الطباعة المركزية ولا يوجد `مركز الطباعة` كقسم مستقل:

```text
كشوف الدعاوى
كشوف الجلسات
كشوف النواقص
كشوف أوامر العمل
كشوف المالية
لائحة عمل اليوم والغد
تصدير PDF
```

### 2.7 إدارة المكتب

بديل `إعدادات النظام` كتسمية للمحامي:

```text
بيانات المكتب
المستخدمون والصلاحيات
سجل المسؤولية
النسخ الاحتياطي
القوائم المرجعية
```

## 3. الأقسام التي تُزال من التنقل الرئيسي

تُزال من الشريط الجانبي النهائي:

```text
الشؤون القانونية
السجلات والجهات
الأشخاص والجهات
الوكالات
الأرشيف الفيزيائي
الموارد والمستندات
المستندات الرقمية
مركز الطباعة كقسم مستقل
إعدادات النظام كتسمية
الإدارة والتقارير كتسمية
```

مع الإبقاء على البيانات والمسارات مؤقتاً حتى يتم نقل الوظائف.

## 4. ترتيب التنفيذ البرمجي المرحلي

### المرحلة 0 — تثبيت خط الأساس

- التأكد من بناء آخر نسخة.
- إصلاح أي أخطاء حالية قبل الهيكلة.
- عدم تعديل سلوك وظيفي.

### المرحلة 1 — إعادة تسمية وتنظيم الشريط الجانبي مؤقتاً دون كسر المسارات

الهدف:
- تغيير التسميات بما يوافق الهيكل النهائي.
- إبقاء المسارات القديمة صالحة.
- عدم إنشاء شاشات جديدة كبيرة بعد.

أعمال المرحلة:
- `الشؤون القانونية` → `ملفات المكتب`.
- `الإجراءات الإدارية` → `الإجراءات والمعاملات`.
- إزالة مجموعة `السجلات والجهات` من الشريط الجانبي الرئيسي أو إخفاؤها مرحلياً حسب توفر البدائل.
- `الموارد والمستندات` تُفكك مرحلياً.
- `الإدارة والتقارير` تُفكك إلى `المالية والصندوق` و`التقارير والكشوف` و`إدارة المكتب`.

### المرحلة 2 — بناء شاشة ملفات المكتب الموحدة

الهدف:
- شاشة واحدة بفلاتر:

```text
جارية / منتهية / تحتاج استكمال
دعاوى / إجراءات / عقود / شركات / وكالات
تصنيفات فرعية لكل نوع
```

### المرحلة 3 — دمج الأشخاص والوكالات داخل ملفات المكتب والمعالجات

- لا تبقى الأشخاص والوكالات كأقسام مستقلة.
- تبقى كبيانات أطراف داخل الملفات.
- تعديل معالجات الإضافة لتدعم اختيار/إضافة الأطراف والوكالات داخلياً.

### المرحلة 4 — مكتب العمل اليومي

- تحويل لوحة اليوم إلى مكتب العمل:

```text
اليوم = إدخال النتائج
الغد = التحضير والتوزيع
الأسبوع = رؤية قريبة
التقويم = عرض زمني
```

- دعم المهام المخصصة وقوالب نتائجها لاحقاً.

### المرحلة 5 — مركز إدخال الأرشيف القديم

- دفعات إدخال.
- أرشيف ورقي/إلكتروني/CSV.
- تمييز الجاري والمنتهي.
- صندوق غير مصنف.
- ملفات جارية تحتاج استكمال.

### المرحلة 6 — المكتبة القانونية

- استقلالها عن المستندات.
- تجهيزها للقوانين والاجتهادات والمراجع.

### المرحلة 7 — النماذج القانونية

- استيراد نموذج.
- إنشاء نموذج.
- توليد مستند من نموذج.

### المرحلة 8 — التقارير والكشوف

- دمج الطباعة المركزية.
- إبقاء طباعة الملفات داخل ملفاتها.

### المرحلة 9 — المالية والصندوق

- إعادة هيكلة مالية الملف وذمم الموكلين والصندوق.

### المرحلة 10 — إدارة المكتب والقوائم المرجعية

- إدارة القوائم المرجعية بإضافة/تعديل/تعطيل/حذف آمن.

### المرحلة 11 — الاختبار الشامل

- اختبار كل شاشة وكل وظيفة.
- اختبار الصلاحيات.
- اختبار Audit.

### المرحلة 12 — المساعد الذكي

- بعد استقرار كل شيء فقط.

## 5. سياسة التنفيذ العملية

كل مرحلة تُنفذ كفرع منطقي داخلي وcommit واضح، ولا نخلط أكثر من نطاق كبير في commit واحد إلا عند الضرورة.

مثال commits:

```text
refactor(nav): align sidebar labels with office structure
feat(files): add office files unified shell
feat(work): add daily work center shell
feat(archive): add archive intake center shell
```

## سجل تنفيذ فعلي — المرحلة 1: تنظيم الشريط الجانبي 2026-07-15

تم تنفيذ المرحلة الأولى من إعادة الهيكلة البرمجية دون كسر المسارات القديمة:

- تغيير مجموعة `الشؤون القانونية` إلى `ملفات المكتب`.
- إضافة `كل الملفات` كمدخل أول ضمن ملفات المكتب عبر المسار الحالي `/files`.
- تنظيم عناصر ملفات المكتب كالتالي:
  - الدعاوى
  - الإجراءات والمعاملات
  - العقود
  - الشركات
  - الوكالات
- إزالة مجموعة `السجلات والجهات` من التنقل الرئيسي، مع إبقاء المسارات القديمة صالحة داخلياً.
- إزالة مجموعة `الموارد والمستندات` من التنقل الرئيسي.
- جعل `المكتبة القانونية` قسماً مستقلاً.
- جعل `النماذج القانونية` قسماً مستقلاً مؤقتاً عبر مسار قوالب العقود الحالي لحين بناء شاشة النماذج القانونية العامة.
- جعل `المالية والصندوق` قسماً مستقلاً.
- جعل `التقارير والكشوف` قسماً مستقلاً ويضم البحث والتقارير والطباعة والتصدير.
- تغيير `إعدادات النظام` إلى `إدارة المكتب`.
- تحديث منطق تحديد العنصر النشط في الشريط الجانبي وفق الهيكل الجديد.
- تحديث فلترة عناصر الشريط حسب الصلاحيات بما يناسب الهيكل الجديد.

> ملاحظة: هذه مرحلة تنظيم تنقل وتسميات فقط. لم يتم حذف أي جداول أو مسارات قديمة، ولم يتم نقل وظائف الأشخاص/الوكالات/الأرشيف الفيزيائي نهائياً بعد. ذلك سيتم في المراحل اللاحقة حسب الخطة.

## سجل تنفيذ فعلي — المرحلة 2: بداية شاشة ملفات المكتب الموحدة 2026-07-15

تم تنفيذ خطوة أولى من المرحلة الثانية دون حذف الشاشات القديمة:

- توسيع مزود `uiFilesProvider` ليجمع ملفات المكتب من مصادر حقيقية متعددة:
  - الدعاوى
  - العقود
  - الشركات
  - الإجراءات والمعاملات
  - الوكالات
- إعادة تسمية شاشة `/files` إلى `ملفات المكتب`.
- تحديث تبويب `عاملة` إلى `جارية` بما يوافق مصطلحات الهيكلة الجديدة.
- جعل بطاقة الملف تفتح المسار الحقيقي حسب نوع الملف:
  - دعوى → `/cases/:id`
  - عقد → `/contracts/:id`
  - شركة → `/companies/:id`
  - إجراء → `/procedures/:id`
  - وكالة → `/poa/:id`
- إبقاء تبويبات الحالة الحالية مؤقتاً لحين بناء الواجهة النهائية ذات طبقات:
  - جارية / منتهية / تحتاج استكمال
  - النوع
  - التصنيف الفرعي

> هذه خطوة تأسيسية لشاشة ملفات المكتب، وليست النسخة النهائية الكاملة. الهدف الحالي هو توحيد مصادر الملفات وفتح كل ملف من مكان واحد دون كسر المسارات الحالية.

### سجل تنفيذ فعلي — تحسين واجهة ملفات المكتب 2026-07-15

تم تحسين واجهة `/files` لتصبح أقرب إلى هيكل `ملفات المكتب` المعتمد:

- استبدال تبويبات الملفات القديمة بواجهة فلاتر واضحة:
  - الملفات الجارية
  - الملفات المنتهية
  - تحتاج استكمال
  - كل ملفات المكتب
- إضافة فلتر نوع الملف:
  - دعوى
  - عقد
  - شركة
  - إجراء إداري
  - وكالة
- إضافة بحث موحد برقم الملف أو الاسم أو المحكمة/الجهة أو رقم الأساس/القيد.
- إضافة شريط ملخص يعرض:
  - المعروض
  - الجارية
  - المنتهية
  - تحتاج استكمال
- إبقاء بطاقة الملف تفتح المسار الحقيقي حسب نوع الملف.
- إبقاء التبويبات القديمة غير مستخدمة مؤقتاً في الكود إلى حين تنظيف نهائي بعد استقرار الهيكلة.
