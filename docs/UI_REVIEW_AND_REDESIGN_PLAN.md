> **التاريخ:** 2026-07-09  
> **الحالة:** مرحلة مراجعة تصميمية ووظيفية قبل أي تعديل برمجي جديد

> **الهدف:** مطابقة ما نُفّذ مع الخطة الأصلية، كشف الانحرافات، ثم إعادة تصميم واجهات التطبيق بشكل عصري ومهني مناسب لبرنامج سطح مكتب قانوني سوري.

## 1. منهجية العمل المعتمدة من الآن وصاعداً

لن يتم تعديل أي شاشة مباشرة قبل المرور بالمراحل الآتية:

1. **قراءة الخطة الأصلية للشاشة** من `docs/EXECUTION_PLAN.md`.
2. **قراءة ملفات التنفيذ الفعلية** الخاصة بالشاشة.
3. **مطابقة بند ببند** بين المطلوب والمنفذ.
4. **تصنيف الأخطاء** إلى:
   - أخطاء وظيفية تمنع سير العمل.
   - أخطاء معمارية تخالف منطق التطبيق.
   - أخطاء تجربة مستخدم وتصميم.
   - أخطاء نصوص، Lookups، أو hardcoded values.
5. **اقتراح تصميم عصري بديل** قبل البرمجة.
6. **توثيق قرار التعديل ومعايير القبول**.
7. بعد موافقة الأستاذ هادي: تنفيذ التعديل، اختبار البناء، تحديث التوثيق، ثم الرفع على Git.

## 2. ملخص البحث التصميمي: الثيمات العصرية المناسبة

### 2.1 الاتجاهات العامة المناسبة للتطبيق

من مراجعة اتجاهات تصميم واجهات 2026 وتطبيقات سطح المكتب الحديثة، الأنسب لتطبيق مكتب محاماة ليس التصميم الاستعراضي، بل تصميم مهني يعتمد على:

- **Minimal Professional UI:** واجهة نظيفة، فراغات واضحة، ألوان قليلة، وتباين قراءة ممتاز.
- **Human-Centric Dashboard:** لوحة قيادة تركز على ما يحتاجه المحامي اليوم: جلسات، مهام، نواقص، ملفات حرجة.
- **Light-first Theme:** لأن التطبيق مكتبي، نصي، قانوني، ويحتاج طباعة وتدقيق طويل؛ الوضع الفاتح أنسب كافتراضي.
- **Optional Dark Mode لاحقاً:** مفيد للعمل الليلي، لكن لا يكون الافتراضي الأول.
- **Bento Grid / Modular Cards:** بطاقات منظمة للملخصات والإجراءات السريعة بدل شبكة عشوائية.
- **Fluent/Mica-like surfaces:** استخدام طبقات خفيفة، ظلال ناعمة، وحدود subtle وليس ألوان داكنة ثقيلة في كل مكان.

### 2.2 الثيم المقترح لتطبيق مكتب المحامي

اسم الثيم المقترح:

> **Syrian Legal Modern Light**

#### الألوان المقترحة

- **Primary Navy مخفف:** `#102A43` أو `#123047` بدل الكحلي شبه الأسود الحالي.
- **Legal Gold هادئ:** `#B9975B` للاختيار، الأيقونات المهمة، وخط الأستاذ.
- **Background:** `#F6F8FA` أو `#F7F8FA`.
- **Surface:** أبيض مع حدود خفيفة `#E5E7EB`.
- **Danger:** أحمر واضح للنواقص فقط، وليس للزينة.

#### ملاحظات على الثيم الحالي

الثيم الحالي فيه نواة جيدة، لكنه يحتاج:

- تخفيف قتامة شريط الملاحة.
- توحيد الظلال والحدود.
- إضافة typography scale واضح.
- تضمين خطوط Cairo/Amiri فعلياً في `pubspec.yaml` بدل ذكرها فقط.
- دعم density مناسب لبرنامج سطح مكتب، لأن واجهات المحاماة تحتاج عرض بيانات كثيرة بدون ازدحام.

## 3. ملخص البحث التصميمي: توزيع التبويبات والملاحة

### 3.1 المبادئ المهنية للملاحة في Desktop

حسب إرشادات Material 3، تطبيقات سطح المكتب لا يُفضّل لها Bottom Navigation، بل الأفضل استخدام:

- **Navigation Rail** للشاشات المتوسطة والكبيرة.
- **Expanded Navigation Rail / Navigation Drawer** للشاشات الواسعة.
- **Tabs داخل الصفحات** فقط للمحتوى الفرعي، وليس للتنقل الرئيسي.

### 3.2 التوزيع المقترح للتطبيق

التوزيع الحالي فيه 6 تبويبات، لكنه يخلط بين “فريق المكتب” و “الأشخاص” وبين “جديد” كصفحة مستقلة. المقترح:

#### الملاحة الرئيسية

1. **لوحة اليوم**
   - مهام اليوم
   - جلسات قريبة
   - ملفات ناقصة
   - أعمال متأخرة

2. **عمل جديد**
   - زر/معالج منظم وليس Grid فقط
   - يبدأ بسؤال: بدء عمل جديد أم أرشفة عمل سابق؟

3. **الأرشيف**
   - دعاوى
   - عقود
   - شركات
   - إجراءات إدارية
   - وكالات
   - مستندات

4. **الأشخاص والجهات**
   - موكلون
   - خصوم
   - محامو خصم
   - كتاب عدل ومندوبون
   - فريق المكتب

5. **البحث والتقارير**
   - بحث شامل
   - طباعة قانونية
   - تصدير PDF

6. **الإعدادات والأمان**
   - بيانات المكتب
   - النسخ الاحتياطي
   - القوائم المرجعية
   - الأمان

### 3.3 مبدأ التبويبات الداخلية

أي شاشة ملف، مثل دعوى أو شركة أو عقد، تستخدم Tabs داخلية. أما الشاشة الرئيسية فتستخدم NavigationRail أو Sidebar فقط.

## 5. خطة تعديل الشاشة الأولى بعد الموافقة

### 5.1 الهدف من التعديل

تحويل الشاشة الرئيسية من Grid بدائي إلى لوحة عمل قانونية احترافية فيها:

- لوحة يومية مفيدة.
- معالج عمل جديد واضح.
- أرشفة عمل سابق.
- ملاحة أكثر عصرية.
- أسلوب بصري مهني.

### 5.2 التصميم المقترح للشاشة

#### الهيكل العام

- **Sidebar/NavigationRail قابل للتوسعة** على اليمين بسبب RTL.
- **Header أعلى المحتوى** يحوي:
  - اسم المكتب.
  - اسم الأستاذ.
  - التاريخ الحالي.
  - زر بحث سريع.
  - زر إشعارات مع عداد.
- **منطقة محتوى رئيسية** بتصميم Bento Cards.

#### محتوى تبويب “عمل جديد”

بدل البطاقات الأربع المباشرة، يظهر:

1. بطاقة كبيرة: **بدء عمل جديد**
2. بطاقة كبيرة: **أرشفة عمل سابق**

بعد الضغط تظهر نافذة اختيار التصنيف:

- دعوى قضائية.
- إجراء إداري.
- شركة.
- عقد.
- وكالة.
- شخص/جهة.
- مستند مستقل.
- مهمة يدوية.

### 5.3 معايير القبول

لا يعتبر تعديل الشاشة مقبولاً إلا إذا تحقق التالي:

- وجود خيارين واضحين: بدء جديد / أرشفة سابق.
- فتح نافذة أو معالج تصنيف واضح.
- دعم RTL دون التفاف أو اختلال.
- عدم وجود نصوص رئيسية مشتتة داخل الكود قدر الإمكان.
- استخدام GoRouter أو توثيق سبب مؤقت لعدم التحويل الكامل.
- Grid متجاوب حسب عرض النافذة.
- وجود مؤشرات على الأقل لـ:
  - مهام اليوم.
  - الملفات الناقصة.
  - الجلسات القريبة.
  - المتأخرات.
- عدم كسر تشغيل Windows.

### 5.4 ما لن يتم تعديله في أول خطوة

حتى لا نخلط المهام، أول تعديل لن يدخل في:

- إعادة بناء كل شاشات الأرشيف.
- تعديل قاعدة البيانات.
- تعديل ملفات الدعوى أو الشركة أو العقد.
- تنفيذ الأمان الكامل.

هذه تبقى لمراحل لاحقة شاشة شاشة.

### 8.2 المطلوب بالخطة الأصلية

وفق الخطة الأصلية، شاشة الأرشيف يجب ألا تكون مجرد تبويبات منفصلة، بل مركز أرشفة وتشغيل يسمح بالوصول السريع والبحث والفلترة في ملفات المكتب، وخاصة:

- عرض ملفات الدعاوى والعقود والشركات والإجراءات والأشخاص والوكالات.
- البحث الفوري والشامل.
- فلاتر مخصصة: ملفات اليوم، ملفات ناقصة، جلسة قريبة، بانتظار استكمال رقم الأساس، ملفات منتهية.
- إظهار الحالة التشغيلية والقانونية لكل ملف.
- دعم الوصول السريع للملف وتفاصيله.

### 8.3 المنفذ فعلياً

- `ArchiveScreen` ينفذ `TabBar` بستة تبويبات: الدعاوى، الإجراءات، الشركات، العقود، الأشخاص، الوكالات.
- كل تبويب يفتح شاشة قائمة مستقلة.
- يوجد بحث محلي داخل بعض الأقسام، مثل الدعاوى والعقود والشركات والإجراءات والأشخاص.
- لا يوجد بحث موحد على مستوى الأرشيف كله.
- لا توجد فلاتر تشغيلية موحدة على مستوى الأرشيف.

### 8.4 نسبة المطابقة الحالية

**45% تقريباً.**

الهيكل العام موجود، لكن منطق الأرشيف الذكي والمركزي غير منفذ.

### 8.6 قرار المراجعة

شاشة الأرشيف تعمل كقوائم أساسية، لكنها لا تطابق مفهوم الأرشيف التشغيلي المطلوب. عند التعديل لاحقاً يجب تحويلها إلى شاشة أرشيف موحدة ذكية.

### 8.7 خطة تعديل الأرشيف لاحقاً

عند الوصول لمرحلة التعديل، يجب تنفيذ ما يلي:

1. بناء شريط بحث موحد يبحث في الدعاوى والعقود والشركات والإجراءات والأشخاص والوكالات.
2. إضافة فلاتر تشغيلية واضحة:
   - الكل.
   - ملفات اليوم.
   - الملفات الناقصة.
   - جلسات قريبة.
   - بانتظار رقم الأساس.
   - ملفات منتهية.
   - متأخرات.
3. إضافة بطاقات إحصائية أعلى الشاشة:
   - عدد الملفات العاملة.
   - عدد النواقص.
   - جلسات الأسبوع.
   - عقود تنتهي قريباً.
   - شركات قيد التأسيس.
   - معاملات متأخرة.
4. إصلاح فلاتر الأشخاص حسب الأدوار الحقيقية من قاعدة البيانات.

### 20.2 نطاق المراجعة

- النسخ الاحتياطي والاستعادة.
- دقة التوثيق مقارنة بالتنفيذ.

### 20.4 قرار المراجعة العامة

المشروع أصبح قابلاً للتشغيل على Windows، وهذا تقدم مهم، لكنه ليس مكتمل وظيفياً كما ادعت تقارير التسليم السابقة. التنفيذ الحالي أقرب إلى نموذج أولي واسع النطاق مع بنية بيانات جيدة جزئياً، لكنه يحتاج مرحلة إعادة بناء واجهات ومنطق ربط فعلية.

### 20.5 خطة العمل العامة لاحقاً

بعد انتهاء مراجعة الشاشات، يجب اعتماد خطة إعادة تنفيذ على مراحل:

1. تحديث التوثيق الرسمي لإزالة ادعاءات اكتمال 100%.
2. اعتماد الثيم والملاحة الجديدة.
3. تنفيذ Dashboard وArchive موحدين.
4. تفعيل الأشخاص والوكالات والمستندات كمراكز بيانات فعلية.
5. إصلاح الدعاوى والمهام والنواقص أولاً لأنها قلب عمل المكتب.
6. تنفيذ المستندات والمالية عبر Repositories الموجودة.
7. إصلاح النسخ الاحتياطي والاستعادة والأمان.
8. توحيد GoRouter وLookups وActivityLog.

### 20.6 معايير قبول عامة قبل إعلان الاكتمال مستقبلاً

- لا توجد شاشات Placeholder.
- كل زر ظاهر ينفذ وظيفة حقيقية.
- كل تبويب معلن موصول بالبيانات الفعلية.
- البحث والأرشيف شاملان فعلياً.
- المستندات والمالية والنواقص تعمل في كل كيان.
- النسخ الاحتياطي والاستعادة آمنان ومجربان.
- التوثيق يعكس الواقع ولا يعلن اكتمالاً غير محقق.
