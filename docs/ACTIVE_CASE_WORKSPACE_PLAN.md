# خطة شاشة الدعوى الجارية — Workspace الدعوى

تاريخ الخطة: 2026-07-17

هذه الخطة تضبط طريقة عرض وإدارة الدعوى الجارية داخل ملفات المكتب، بعد الاتفاق على أن `الملفات الجارية` هي شاشة تعامل وبحث وتعديل، وأن إنشاء الملفات الجديدة يتم فقط من `مكتب العمل > إنشاء جديد`.

---

## 1. قاعدة رقم الملف ورقم الأساس

يجب الفصل دائماً بين:

```text
رقم ملف المكتب
رقم الأساس القضائي
```

- `رقم ملف المكتب`: رقم داخلي لإدارة الملف داخل المكتب والتطبيق.
- `رقم الأساس`: رقم قضائي مرتبط بالمحكمة أو المرحلة القضائية.

لا يجوز تسمية رقم ملف المكتب بـ `رقم الدعوى` حتى لا يختلط مع رقم الأساس.

---

## 2. رأس ملف الدعوى الجاري

يعرض رأس ثابت أعلى شاشة الدعوى الجارية في كل التبويبات.

البيانات:

```text
رقم ملف المكتب
نوع الدعوى
صفة / نوع المرحلة الحالية
المحكمة / المرجع
رقم الأساس وسنته للمرحلة الحالية
الموكل الرئيسي
الخصم الرئيسي
الموعد القادم
حالة الدعوى
المحامي المسؤول
المحامون المساعدون إن وجدوا
```

يبقى الرأس ظاهراً أثناء التنقل بين تبويبات الملف.

---

## 3. الأدوار الأساسية داخل التطبيق

يجب التمييز بين:

```text
دور الصلاحيات
الصفة الوظيفية / المهنية داخل المكتب
```

### 3.1 دور الصلاحيات

يحدد ماذا يستطيع المستخدم فعله في النظام.

أمثلة:

```text
مالك المكتب
مدير النظام
مدير مكتب
محامي
محامي مساعد
سكرتير / أمين سر
محاسب
مدخل بيانات / أرشيف
مراجع / مدقق
مشاهد فقط
```

### 3.2 الصفة المهنية داخل الملفات

تحدد علاقة المستخدم بملف معين.

أمثلة:

```text
محامي مسؤول
محامي مساعد
موظف متابع
مدخل نتيجة
مدقق مستندات
```

### 3.3 قرار الدعوى الجارية

في الدعوى الجارية نعتمد:

```text
محامٍ مسؤول واحد
محامون مساعدون اختياريون
```

المحامي المسؤول يجب أن يختار من مستخدمي التطبيق، لا كنص حر.

يعرض:

```text
رقم ملف المكتب
تاريخ فتح الملف
نوع الدعوى
المرحلة الحالية
صفة المرحلة الحالية
المحكمة الحالية
رقم الأساس وسنته

الطلبات
آخر إجراء
الموعد القادم
الإجراء القادم
المحامي المسؤول
المحامون المساعدون

```

```text
تعديل البيانات الأساسية

```

زر الإغلاق أو إنهاء المرحلة يفضّل أن يكون في تبويب المراحل والجلسات، وليس في كل مكان.

## 7. تبويب الأطراف والوكالات

يدعم:

```text
عدة موكلين
عدة خصوم
صفات متعددة للأطراف
وكالات متعددة
```

صفات الأطراف المقترحة:

```text
مدعي
مدعى عليه
مستأنف
مستأنف عليه
طاعن
مطعون ضده
مدخل
متدخل
طالب مخاصمة
مخاصم ضده
صفة مخصصة
```

### الوكالات

الوكالة ترتبط مبدئياً بالدعوى، ومعها بيانات الموكل أو الموكلين المشمولين بها قدر الإمكان.

بيانات الوكالة:

```text
نوع سند التوكيل
رقم سجل الوكالة
الرقم الأبيض
تاريخ ووقت التنظيم
فرع النقابة
مندوب الوكالات
هاتف المندوب
صورة السند
ملاحظات / نطاق الوكالة
```

## 8. تبويب المراحل والجلسات

### 8.1 المراحل

كل مرحلة يجب أن تحمل صفة مركبة، مثل:

```text
صلح مدني
بداية مدنية
استئناف مدني
نقض مدني
صلح جزاء
بداية جزاء
استئناف جنح
جنايات
نقض جزائي
محكمة قضاء إداري
المحكمة الإدارية العليا
```

أي أن المرحلة لا تكون `صلح` فقط، بل يجب أن تتضمن الاختصاص أو النوع.

### 8.2 بيانات المرحلة

كل مرحلة تحتوي:

```text
نوع / صفة المرحلة
المحكمة / المرجع
المحافظة
رقم الغرفة
رقم الأساس أو رقم المرجع
السنة
تاريخ البدء
تاريخ الانتهاء
القرار / النتيجة
الوثائق المرتبطة
```

### 8.3 تحذير الحقول الثابتة

بعض الحقول يجب التعامل معها كحقول ثابتة بعد الحفظ أو تعديلها ضمن آلية خاصة فقط.

أمثلة:

```text
نوع الدعوى
نوع المرحلة
المحكمة / المرجع
رقم الأساس وسنته
رقم ملف المكتب
```

عند إدخال هذه الحقول يجب عرض تنبيه واضح مثل:

```text
تنبيه: هذا الحقل يؤثر على بنية الملف والمراحل القضائية، وقد لا يمكن تعديله لاحقاً إلا من شاشة تعديل المرحلة.
```

### 8.4 الجلسات داخل المرحلة

الجلسات تعرض داخل المرحلة التي تنتمي إليها، فلا تختلط جلسات البداية بجلسات الاستئناف أو النقض.

كل جلسة:

```text
تاريخ الجلسة
وقت الجلسة
الحضور
ما تم في الجلسة
قرار المحكمة
الإجراء المطلوب لاحقاً
الموعد القادم
ملاحظات
مرفقات
```

### 8.5 نتيجة الجلسة

حقول نتيجة الجلسة:

```text
تاريخ الجلسة
هل حضر الموكل؟
هل حضر الخصم؟
هل حضر وكيل الخصم؟
ما تم في الجلسة
قرار المحكمة
الإجراء المطلوب لاحقاً
الموعد القادم
مرفقات
ملاحظات
```

الإجراء القادم يبقى نصاً حراً مع اقتراحات جاهزة.

اقتراحات الإجراء القادم:

```text
تبليغ
مرافعة
إبراز وثائق
سماع شهود
خبرة
تفهيم حكم
رد على مذكرة
تدقيق
إمهال
مخصص
```

## 9. إغلاق / إنهاء مرحلة

تعتمد التسمية:

```text
إغلاق / إنهاء مرحلة
```

وليس فقط `إغلاق الدعوى`، لأن الدعوى قد تغلق مرحلة وتبدأ مرحلة طعن لاحقة.

### أسباب الإغلاق

```text
التخلي عن الملف / تسليمه
إسقاط / تنازل / صلح
صدور قرار قضائي
انتهاء التنفيذ
سبب آخر
```

### التخلي عن الملف

عند اختيار التخلي:

```text
تاريخ التخلي
اسم المحامي الجديد إن وجد
هل تريد إنشاء لا مانع؟
ملاحظات
```

صيغة لا مانع يجب أن تكون قابلة للتعديل قبل التصدير.

### إسقاط / تنازل / صلح

حقول:

```text
من قام بالإسقاط؟
نوع الإسقاط
تاريخ الإسقاط
مستند الإسقاط
ملاحظات
```

### صدور قرار قضائي

حقول:

```text
رقم القرار
تاريخ القرار
مضمون القرار
إرفاق القرار
```

ثم يختار المستخدم الأثر:

```text
إغلاق نهائي
فتح استئناف
فتح نقض
فتح مخاصمة
فتح تنفيذ
```

حسب المرحلة الحالية.

## 10. تبويب المستندات

يعرض مستندات الدعوى بقائمة واحدة مع فلاتر، لا بتبويبات كثيرة.

فلاتر مقترحة:

```text
وكالات
استدعاءات
تبليغات
ضبوط
قرارات
مذكرات
إيصالات
أخرى
```

يدعم:

```text
إضافة مستند
فتح مستند
تعديل بيانات الأصل الورقي
```

---

## 11. تبويب أوامر العمل

يعرض أوامر العمل المرتبطة بالدعوى:

```text
مفتوحة
نتيجة مدخلة
معتمدة
مؤجلة
ملغاة
تحتاج تصحيح
```

أمر العمل المعتمد يمكن أن يحدث الملف حسب نتيجته:

- إضافة مستند.
- إضافة مصروف.
- إضافة موعد لاحق.
- إضافة حدث في Timeline.
- إغلاق نقص عند الاقتضاء.

---

## 12. تبويب المالية

يعرض:

```text
اتفاقات الأتعاب
سندات القبض
المصاريف
الرصيد
```

يدعم حسب الصلاحية:

```text
إضافة اتفاق أتعاب
إضافة سند قبض
إضافة مصروف
طباعة كشف مالي للملف
```

مع الحفاظ على القاعدة:

```text
اتفاق أتعاب ≠ سند قبض
```

---

## 13. تبويب النواقص

يعرض:

```text
نواقص مفتوحة
نواقص محلولة
نواقص متجاهلة
```

النواقص تحذيرية ولا تمنع الحفظ، إلا إذا تقرر لاحقاً منع إجراء محدد.

أمثلة النواقص:

```text
لا توجد وكالة
لا يوجد رقم أساس
لا يوجد موعد قادم
مستند مطلوب غير مرفق
أصل ورقي غير محفوظ
لم يتم إدخال نتيجة جلسة اليوم
```

## 14. تبويب سجل الأحداث

Timeline كامل، ويشمل:

```text
فتح الملف
إضافة وكالة
إضافة جلسة
إدخال نتيجة
إضافة مستند
أمر عمل
قبض
مصروف
إغلاق مرحلة
استئناف
نقض
مخاصمة
```

يمكن لاحقاً إضافة فلاتر للـ Timeline، لكن البداية تكون عرضاً كاملاً.

```text
الإجراء القادم
```

### تعدل من تبويب المراحل فقط

```text
المحكمة
```


## 17. الخلاصة التنفيذية

تعتمد شاشة الدعوى الجارية على:

```text
رأس ثابت
تبويبات متخصصة
مراحل تحمل صفة مركبة
جلسات داخل كل مرحلة
إغلاق/إنهاء مرحلة وليس مجرد إغلاق دعوى
رقم ملف مكتب منفصل عن رقم الأساس
محام مسؤول من مستخدمي التطبيق
نواقص تحذيرية
طباعة من قائمة واحدة
```

## 18. قرارات إضافية مثبتة بعد النقاش

### 18.1 المحامي المسؤول

- المحامي المسؤول يجب أن يختار من مستخدمي التطبيق.
- نحتاج لاحقاً إلى تسمية أدوار مهنية أساسية بجانب أدوار الصلاحيات.
- تعتمد الدعوى محامياً مسؤولاً واحداً، مع إمكانية محامين مساعدين اختيارياً.

### 18.2 حالة الدعوى

تم اعتماد الحالات التفصيلية المقترحة:

```text
قيد التحضير
مسجلة
قيد التبليغ
قيد المحاكمة
بانتظار قرار
بانتظار تنفيذ
معلقة
مؤجلة
مغلقة / منتهية
```

### 18.3 صفة المرحلة

لا يكفي أن تكون المرحلة `صلح` أو `بداية` فقط، بل يجب أن تحمل صفة الاختصاص:

```text
صلح مدني
بداية مدنية
صلح جزاء
بداية جزاء
استئناف مدني
نقض جزائي
محكمة قضاء إداري
```

### 18.4 الحقول الثابتة

الحقول التي تؤثر على بنية الملف أو المرحلة يجب أن تعرض تحذيراً قبل الحفظ:

```text
نوع الدعوى
نوع المرحلة
المحكمة / المرجع
رقم الأساس
سنة الأساس
رقم ملف المكتب
```

وتعدل لاحقاً من تبويب المراحل لا من الملخص.

### 18.5 إغلاق / إنهاء مرحلة

تم اعتماد تسمية:

```text
إغلاق / إنهاء مرحلة
```

بدلاً من `إغلاق الدعوى` فقط، لأن الإغلاق قد يكون نهاية مرحلة لا نهاية الملف كاملاً.

### 18.6 أسباب الإغلاق

```text
التخلي عن الملف / تسليمه
إسقاط / تنازل / صلح
صدور قرار قضائي
انتهاء التنفيذ
سبب آخر
```

### 18.7 التخلي ولا مانع

عند التخلي عن الملف يجب دعم تصدير كتاب `لا مانع` قابل للتعديل قبل الطباعة أو الحفظ.
