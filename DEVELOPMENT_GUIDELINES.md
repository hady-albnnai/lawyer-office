## 🔄 دورة حياة التعديل والتطوير (Workflow)
يُمنع إغلاق أي مهمة أو إبلاغ المستخدم بالانتهاء إلا بعد استكمال الخطوات الثلاث التالية بالترتيب الدقيق:

### 1️⃣ المرحلة البرمجية (Coding Phase)
- تنفيذ التعديلات المطلوبة في ملفات كود Dart (`.dart`) ومخططات قاعدة البيانات (`Drift / SQLite`).
- التأكد المطلق من عدم وجود أخطاء برمجية (Syntax & Type Errors) وتشغيل أدوات التحليل (`flutter analyze`).
- اختبار المنطق التجاري والقانوني (Business & Legal Logic) لضمان عدم وجود ثغرات في دورة الحياة الموحدة للعمليات، الترقيم السنوي الآمن، الماليّة الموحدة، أو نظام النواقص التلقائي.

### 2️⃣ المرحلة التوثيقية (Documentation Phase) ⚠️ **(إلزامية)**
- مراجعة ملف `docs/EXECUTION_PLAN.md`: إذا تم تغيير أي حقل في قاعدة البيانات، تعديل هندسي، أو إضافة ميزة جديدة، يجب تحديث المواصفات ووثيقة التنفيذ فوراً.
- مراجعة وتحديث حالة المهام في خطة التنفيذ (نقل الميزة من "قيد التنفيذ" أو "مجدولة" إلى "مكتملة") بعد التأكد التام من عملها.
- توثيق أي تغييرات في الجداول المرجعية، المحركات الخلفية (Engines)، أو إعدادات الطباعة القانونية.

### 3️⃣ مرحلة الرفع (Deployment Phase) ⚠️ **(إلزامية)**
- تنفيذ سلسلة أوامر الرفع لضمان وصول التعديلات إلى المستودع مباشرة في نهاية كل مهمة:
  ```bash
  git add .
  git commit -m "وصف دقيق للتعديلات المنجزة ومطابق الخطة"
  git push origin main
  ```

## 🚫 محظورات تقنية
- **لا** تترك أي `print` أو `debugPrint` في الكود النهائي المنتظم (استخدم نظام السجلات `ActivityLog` أو أدوات المراقبة المعتمدة).
- **لا** تضف نصوصاً ثابتة (Hardcoded) في الواجهات؛ كل العناوين والقوائم القانونية يجب أن تمر عبر الجداول المرجعية (Lookups) أو ملفات الثوابت والتوطين.
- **لا** تغير في بنية الجداول دون تحديث مخطط قاعدة بيانات Drift (`schema.dart`) وتشغيل `build_runner` لضمان توافق الأكواد المولدة (`database.g.dart`).
- **لا** تتجاوز قاعدة "دورة الحياة الموحدة" لأي كيان تشغيلي (`scheduled`, `in_progress`, `completed`, `postponed`, `cancelled`).
- **لا** تحذف المرفقات أو الملفات المرتبطة بالكيانات دون احترام قواعد القيود الخارجية (`ON DELETE RESTRICT` أو `CASCADE` حسب التصميم المعتمد).

## 🎯 مبدأ الدقة المتناهية (إلزامي — مقتبس ومحفوظ من دستور العقارات ومُطبق بصرامة هنا)

كل مهمة تُنفَّذ بدقة متناهية وتحليل كامل مهما أخذت من وقت وجهد، **دون اختصار ودون حلول جزئية**. السببان الموجبان:

1. **سبب يخص المشروع:** النظام يتعامل مع ملفات دعاوى، مواعيد جلسات قانونية حساسة، وأمور مالية تخص مكتب المحاماة — أي خطأ في الحسابات، الترقيم السنوي، أو مواعيد الجلسات يسبب أضراراً قانونية حقيقية، فلا مجال لتعديلات متسرعة أو غير مكتملة.
2. **سبب يخص كفاءة العمل:** الحلول الجزئية والاختصارات تجبر صاحب المشروع على فتح محادثات جديدة لنفس الموضوع وإعادة شرح السياق من الصفر — وهذا استنزاف للموارد والوقت دون داعٍ. الحل الكامل من أول مرة أوفر دائماً.

### قواعد التطبيق العملي لهذا المبدأ:
- **لا افتراضات حول مخطط قاعدة البيانات المحلي (`Drift / SQLite / SQLCipher`):** قبل أي تعديل يمس الجداول أو العلاقات، يُتحقق من تناغم تعريفات Drift (`schema.dart`)، القيود الخارجية، الفهارس، وحالة توليد الأكواد.
- **تتبع كامل للتبعيات (Full Dependency Tracking):** عند تعديل أي كيان (Entity)، دالة، أو حقل قاعدة بيانات، تُحصر كل الجهات المستدعية (الجداول المترابطة، DAOs، Repositories، Riverpod Providers، الشاشات، ومحركات النواقص والخط الزمني) وتُحدَّث جميعها في نفس المهمة.
- **معالجة جذر المشكلة لا عرضها:** إن كشف التحليل ثغرات مجاورة في نفس النطاق، تُذكر صراحة في التقرير ويتم التعامل معها فوراً أو إدراجها ضمن المهام ذات الأولوية القصوى — لا تُترك مكتومة.
- **لا تنفيذ دون تصريح واضح للمهام المحورية:** أي تعديل معماري كبير يتطلب توضيح خطوة بخطوة لما سيتم إنجازه والالتزام بالمخطط المعتمد.
- **الإنجاز الحقيقي فقط:** لا يُعلن اكتمال ميزة إلا بعد اكتمال سلسلتها كاملة (`Database Schema` + `Service/Engine` + `Repository` + `Riverpod Provider` + `UI/Screens` + `Documentation` + `Git Push`) وتحديد ما تم إنجازه بدقة متناهية.
